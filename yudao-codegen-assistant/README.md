  ### 📉 优化前 vs 优化后：触目惊心的对比
  
  为了验证效果，我用同一个需求（“新增一个卫星任务接入功能”）分别测试了普通模式和我的“极简模式”。
  
  #### ❌ 优化前：典型的“注水”代码
  
  AI 输出了 **1200+ Token**。
- **前 20 行**：全是 `import java.util...` `import org.springframework...`
- **中间 50 行**：DO 类里全是 `public String getName() { return name; }`

  #### ✅ 优化后：我的 `yudao-pro-saver`
  
  AI 仅输出了 **400 Token**（**节省 67%**）。
- **没有 Import**：默认你知道要导什么包。
- **没有 Getter/Setter**：直接 `@Data` 完事。
- **CRUD 变注释**：标准增删改查直接显示 `// 继承 BaseMapperX`。

  ### 🛠️ 核心秘籍：我是如何“调教”这个 Skill 的？
  
  这个 Skill 的强大，不在于它写了多少代码，而在于它**“不写”**什么。我在提示词中植入了以下**4 条铁律**：
  
  #### 1. Lombok 与 MapStruct 的“智能省略”
  
  > **指令**：所有实体类强制使用 `@Data @Builder`。除非字段名不一致，否则禁止输出 MapStruct 的 `@Mapping` 注解。
  > AI 不再像个复读机一样列出所有字段映射，只有当数据库字段 `user_name` 对应 Java 字段 `name` 时，它才会输出注解。
  
  #### 2. 逻辑“伪代码化”
  
  对于简单的业务判断，我不需要它写完整的 `if-else` 结构。
- **以前**：它写 10 行 Java 代码判断状态。
- **现在**：它直接输出 `逻辑：若状态为“待接入”，则允许规划；否则报错。`
  开发者一眼就懂，不需要阅读语法糖。
  
  #### 3. SQL 的“按需裁剪”
  
  针对 yudao 框架，`create_time`、`updater` 这些字段是基类自带的。
  **优化后的 Skill 会自动省略这些通用字段的定义**，只关注业务特有的字段（如 `satellite_id`, `orbit_param`）。
  
  #### 4. 强制自检机制
  
  在输出前，AI 会自我审查：
- “这行代码是 IDE 自动生成的吗？” -> **删掉。**
- “这个 Import 是 java.lang 下的吗？” -> **删掉。**
